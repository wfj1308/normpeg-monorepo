> 版本：v1.0  
> 目标：把现有 PRD / 架构讨论 / 协议设计 / API 设计 / 产品矩阵，统一收敛为一份可直接使用的 Markdown 架构总文档。  
> 适用对象：产品负责人、CTO、架构师、解决方案顾问、投资沟通。

# 0. 一句话定义

**LayerPeg / NormRef = 一个以“可执行规范”为核心、以“工程状态协议”为底座、以“AI 交互”为入口的工程执行操作系统。**

它要解决的不是“如何更好地搜索 PDF”，而是：

- 如何把规范从静态文本变成可执行程序
- 如何把项目从静态数据表变成状态机系统
- 如何让 AI 只能在规范约束内表达，而不能脱离规则胡乱判断
- 如何把表单、判定、签字、证据、报告统一到一套协议里

---

# 1. 设计目标与核心原则

## 1.1 系统设计目标

1. **规范程序化**：把 PDF 规范、条款、表格、公式、试验方法编译成可执行构件。
2. **判定确定性**：把“合格/不合格/特批”从人工经验转为引擎裁决。
3. **项目状态化**：把工程对象、桩号、段落、合同、变更统一建模为状态机。
4. **过程可追溯**：所有计算、判定、签名、审批都可形成 Proof。
5. **AI 接口化**：大模型负责理解和表达，不负责最终计算和判定。
6. **服务平台化**：底层能力可以通过 API / SDK / 产品矩阵复用。

## 1.2 核心原则

- **规范 = 程序，不是文本**
- **AI = 接口，不是法官**
- **工程 = 状态机，不是表格集合**
- **文档 = 协议对象，不是附件**
- **执行 = 可验证过程，不是黑箱结果**

# 2. 架构总览（四个互补视角）

整套体系不能只用一张图理解。它至少有四个互补视角：

1. **五层模型（产品 / PRD 视角）**：从协议到底层到交互层，适合讲系统定位。
2. **八层运行图（系统实现视角）**：把用户、服务、存储、API 都纳入进来。
3. **文档协议五层（对象表达视角）**：描述任意 LayerPeg 文档的内部结构。
4. **端到端执行闭环（运行过程视角）**：描述用户请求如何变成结果与 Proof。

这四种视角互相嵌套，不互相替代。


## 3.1 第1层：协议与状态层（Protocol / State）

### 定义
定义“工程世界如何存在”的底层规则。

### 关键对象
- `v://`：统一寻址协议
- `ProjectUTXO`：项目主权状态根
- `Fork / Split / Merge`：工程版本管理
- `Space Slot / Space Container / Volume Container`：空间与工程量建模
- `Mapping Kernel`：空间映射内核

### 作用
- 给所有对象一个可寻址、可追溯、可分叉的存在方式
- 让项目、里程段、构件、合同、检测记录都能处在同一状态系统里
- 为执行引擎提供统一上下文

### 一句话
**这是系统的“操作系统内核”。**

---

## 3.2 第2层：规范模型层（Spec Layer）

### 定义
把规范从 PDF / 文本转成可执行模型。

### 关键对象
- `NormDoc`：结构化规范
- `SpecIR`：规范中间表示
- `Component`：执行构件（Input + Path + Gate + State + Proof）
- `SPU`：运行态执行单元（Form + Path + Rule + Proof）
- `Override`：项目/企业/地方标准覆盖

### 作用
- 把条文、表格、公式、试验方法转成程序描述
- 支持版本管理、增量更新、术语映射
- 为执行引擎提供可运行的标准库

### 一句话
**这是“规范编译器 + 构件标准库”。**

---

## 3.3 第3层：执行引擎层（Execution Core）

### 定义
执行规范、驱动状态、生成证据的确定性系统。

### 关键模块
- `Component Registry`
- `Path Engine`
- `Gate Engine`
- `State Engine`
- `Proof Engine`
- `Output Composer`

### 作用
- 把输入数据转成计算结果
- 把结果转成判定
- 把判定转成状态变迁
- 把全过程固化成可验证 Proof

### 一句话
**这是系统真正的“因果执行大脑”。**

---

## 3.4 第4层：业务应用层（Applications）

### 定义
面向角色和场景，把底层能力封装为产品。

### 产品矩阵
- `PegBot`：规范问答、判定解释、技术交底、检查清单
- `FormPeg`：表单执行、自动计算、自动判定、签字归档
- `MarkUnit`：Markdown 编辑器、方案编写、AI 辅助
- `PegUnit`：案例库、Fork 复用、Diff 对比
- `CSD Engine`：施工组织设计自动编排
- `CLI`：工程师终端执行入口

### 一句话
**这是“工程业务应用层”，不是判定层。**

---

## 3.5 第5层：交互与 AI 层（Interface）

### 定义
用户与系统交互的入口层。

### 关键能力
- `NL2Gate`：自然语言 -> 执行请求
- 对话系统：多轮上下文、解释生成、报告生成
- 辅助写作：MarkUnit / PegUnit 侧边栏
- 命令交互：CLI / API 调用

### AI 边界
AI 只做：
- 理解语言
- 补全参数
- 组织解释
- 生成报告

AI 不做：
- 最终数学计算
- 最终规范判定
- 绕过 Gate 直接给结论

### 一句话
**这是“人机交互层”，不是规则执行层。**

## 4.2 八层的关键理解

### 0. 用户 / 外部参与方层
系统要服务的不只是施工员，也包括监理、总工、设计方、业主、法务，以及 BIM / ERP / IoT 等外部系统。

### 1. 交互与产品入口层
把能力暴露出去：网页、表单、编辑器、CLI、API。

### 2. AI 理解与编排层
负责自然语言、术语、输出组织，是“秘书层”。

### 3. 执行引擎层
负责算、判、流转、存证，是“执行中枢”。

### 4. 规范模型与知识资产层
负责保存“规范源代码”和构件标准库。

### 5. 协议与状态内核层
负责项目主权、版本分支、桩号空间和状态模型。

### 6. 文档协议五层
负责统一表达所有可执行文档对象。

### 7. 存储与锚定层
负责对象持久化、历史快照、Merkle Root、外部锚定。

### 8. 服务/API 矩阵层
负责把所有能力以服务方式暴露给产品和第三方系统。

# 5. 架构图三：文档协议五层（对象视角）

```text
LayerPeg Document
├── Header（身份与主权）
├── Gate（裁决与执行策略）
├── Body（业务载荷）
├── Proof（证据链）
└── State（生命周期）
```

## 5.1 Header（身份与主权）

### 定义
定义这个对象“是谁、属于谁、在哪个空间里、引用哪套规范”。

### 常见字段
- `v://` 地址
- 项目 ID
- RootRef
- DID（执行人/签字人）
- NormRef / 规范版本引用
- 时间上下文

### 作用
- 把对象纳入主权项目空间
- 确定版本和上下文
- 支撑后续 Proof 与审计

---

## 5.2 Gate（裁决与执行策略）

### 定义
定义如何判定通过、阻断、特批，以及下一步动作。

### 常见内容
- 单点检查规则
- 聚合检查规则
- 强制阻断条件
- Override 条件
- 异常路径
- 待办动作生成

### 作用
- 决定是否 PASS / BLOCK / CRITICAL / OVERRIDE
- 决定失败后的整改路径

---

## 5.3 Body（业务载荷）

### 定义
承载真实输入、输出和中间计算数据。

### 常见内容
- DTO 输入数据
- SPU 实例数据
- 测试原始值
- 计算过程中的中间值
- 输出结果

### 作用
- 承载“这次执行到底算了什么”

---

## 5.4 Proof（证据链）

### 定义
承载这次执行的证据。

### 常见内容
- 输入哈希
- 计算轨迹
- Gate 决策记录
- 时间戳
- 签名
- 外部锚定地址

### 作用
- 审计
- 追责
- 法务核验
- 结果溯源

---

## 5.5 State（生命周期）

### 定义
承载当前对象所处的状态与历史变迁。

### 常见状态
- `DRAFT`
- `COMPUTED`
- `VALIDATED`
- `QUALIFIED`
- `REJECTED`
- `OVERRIDDEN`
- `ARCHIVED`

### 作用
- 支撑流程推进
- 支撑待办动作
- 支撑状态回放与版本追溯

---

## 5.6 五层与系统分层的关系

- Header -> 协议与状态内核层
- Gate -> 执行引擎层（Gate Engine）
- Body -> 规范模型层 + 执行层 DTO
- Proof -> Proof Engine + 存储与锚定层
- State -> State Engine + 项目状态库

**因此，文档五层是“对象表达格式”，系统五层/八层是“运行架构”。**

# 6. 规范模型架构：NormDoc -> SpecIR -> Component -> SPU

```text
PDF / 原始规范
    ↓
NormDoc（结构化）
    ↓
SpecIR（中间表示）
    ↓
Component Catalog（构件目录）
    ↓
SPU（运行态执行单元）
    ↓
表单 / 计算 / 判定 / Proof
```

## 6.1 NormDoc

### 定义
把 PDF 中的章节、条款、表格、公式、引用关系结构化。

### 作用
- 作为机器可读规范层
- 保留条款级引用能力
- 为 SpecIR 提供原始结构素材

---

## 6.2 SpecIR

### 定义
规范中间表示，是连接“人类规范文本”和“机器可执行逻辑”的桥梁。

### 三层结构
- **语义层**：名称、条号、定义、术语
- **逻辑层**：输入、Path、Gate、State
- **证据层**：Proof 要求、签名要求、锚定要求

### 作用
- 成为规范源代码
- 成为长期可复用资产
- 成为 AI 的可信知识底座

---

## 6.3 Component

### 标准定义

```text
Component = Input + Path + Gate + State + Proof
```

### 含义
- `Input`：输入接口与类型约束
- `Path`：计算 / 转换 / 查表 / 聚合步骤
- `Gate`：判定规则与异常路径
- `State`：生命周期
- `Proof`：执行证据

### 类型示例
- WorkItem：分项工程
- MeasuredItem：实测项目
- TestMethod：试验方法

---

## 6.4 SPU

### 标准定义

```text
SPU = Form + Path + Rule + Proof
```

### 含义
- `Form`：输入表单与字段定义
- `Path`：公式执行
- `Rule`：规则判定
- `Proof`：结果固化

### 作用
- 是系统真正的运行单元
- 决定界面表单、计算过程、规则检查、签字与归档

---

## 6.5 Override

### 定义
项目、企业、地方标准对原规范的覆盖层。

### 作用
- 支持同一构件在不同项目上下文里执行不同标准
- 支持设计变更、特批和本地适配

---

# 7. 执行引擎架构：LayerPeg Core

```text
输入 / 调用
   ↓
Component Registry
   ↓
Path Engine
   ↓
Gate Engine
   ↓
State Engine
   ↓
Proof Engine
   ↓
Output Composer
```

## 7.1 Component Registry

### 定义
根据项目上下文、构件目录、规范版本定位具体可执行单元。

### 作用
- 找到正确构件
- 应用正确 Override
- 挂载正确输入输出模型

---

## 7.2 Path Engine

### 定义
执行数学计算、查表、转换、聚合。

### 作用
- 保证计算确定性
- 把原始试验数据转成可判定结果

### 例子
湿密度 -> 干密度 -> 压实度 -> 标准值查表

---

## 7.3 Gate Engine

### 定义
根据规则裁决结果，并决定是否阻断、特批、进入下一状态。

### 输出类型
- `PASS`
- `BLOCK`
- `CRITICAL`
- `OVERRIDE`

### 作用
- 反幻觉核心
- 合规性判断核心
- 流程推进核心

---

## 7.4 State Engine

### 定义
管理执行对象的生命周期。

### 作用
- 推进流程
- 记录状态历史
- 触发待办事项

---

## 7.5 Proof Engine

### 定义
把输入、计算、判定、签名和锚定组织成证据链。

### 作用
- 形成审计对象
- 支撑法务 / 仲裁 / 回放

---

## 7.6 Output Composer

### 定义
把执行结果输出成不同格式。

### 输出形式
- JSON
- Markdown
- 报告
- 检查清单
- API 响应

---

# 8. 协议与状态内核：v:// + UTXO + Space / Volume

## 8.1 v:// 统一寻址协议

### 标准形式

```text
v://{project-id}/{stake-range}?version={hash}&layer={layer}&time={timestamp}
```

### 作用
- 统一表达项目、空间位置、版本、层位和时间
- 成为所有对象的主权地址

---

## 8.2 ProjectUTXO

### 定义
把项目视为一个独立状态机根，而不是数据库记录。

### 核心思想
- 每个输出代表一个当前仍有效的状态单元
- 状态更新通过“花费旧输出，生成新输出”完成
- 支持全链路追踪和版本控制

---

## 8.3 Fork / Split / Merge

### Fork
用于设计变更、试验变更、并行方案分支。

### Split
用于合同段拆分、分包、里程切片、工程量分解。

### Merge
用于把分支结果合并回主线，并保留历史。

### 作用
把 Git 思想引入工程世界。

---

## 8.4 空间模型

### Space Slot
纯粹的地理地址对象，只负责坐标、桩号、高程等定位。

### Space Container
规范执行实例，是“住在某个空间槽上的执行程序”。

### Volume Container
物理工程量边界，承载体积、数量、几何边界和结算属性。

### 三者关系
- Slot：地址
- Container：执行实体
- Volume：工程量边界

---

## 8.5 Mapping Kernel

### 定义
把“空间键（如桩号）”映射到容器、工程量、规范、状态、待办和 Proof。

### 作用
实现“桩号 -> 一切”。

# 10. 端到端执行闭环图

```text
用户输入 / 表单填报 / API 调用 / 上传 PDF
                    │
                    ▼
            NL2Gate / 参数抽取 / 语义定位
                    │
                    ▼
          v:// 定位项目 + 桩号 + 版本 + 层位
                    │
                    ▼
             Component Registry 定位构件
                    │
                    ▼
                  Path 计算
                    │
                    ▼
                  Gate 裁决
           ┌────────┴─────────┐
           │                  │
         PASS          BLOCK / OVERRIDE
           │                  │
           ▼                  ▼
      State 更新        生成异常路径 / 待办
           │                  │
           └────────┬─────────┘
                    ▼
                 Proof 生成
                    │
                    ▼
     JSON / Markdown / 报告 / 清单 / API 响应输出
                    │
                    ▼
         回写状态库 / Mapping / 外部锚定 / 审计
```

## 10.1 闭环解释

1. 用户通过对话、表单、CLI 或 API 发起请求。
2. NL2Gate 识别意图、抽取参数。
3. 系统通过 v:// 确定项目上下文。
4. Registry 找到对应构件或 SPU。
5. Path 执行计算。
6. Gate 做最终裁决。
7. State 记录生命周期变化。
8. Proof 记录全过程。
9. Output Composer 组织不同输出形式。
10. 结果回写到状态系统并可被再次查询。

# 14. 空间-体积双容器架构

## 14.1 核心哲学

一个工程对象（如一段路基、一根桩基）必须同时具备两种边界：

1. **空间容器**：规范执行边界
2. **体积容器**：工程量与结算边界

两者结合，才能形成：
- 可执行边界
- 可结算边界
- 可追溯边界

---

## 14.2 结构图

```text
Space Slot（地址）
   └── Space Container（规范 + 状态 + 人员）
           └── Volume Container（体积 + 工程量 + BOQ）
```

## 14.3 作用

- Space Slot：提供地理索引
- Space Container：提供规范执行实例
- Volume Container：提供结算与工程量实体

# 18. 为什么这套架构成立

## 18.1 解决了传统 RAG 做不到的问题

- RAG 只能找段落，不能执行规范
- RAG 不能做确定性数学和状态管理
- RAG 不能天然形成审计级证据链
- RAG 不能做工程分支与版本控制

## 18.2 解决了传统工程信息系统做不到的问题

- 传统系统有表，没有规范执行内核
- 传统系统有数据，没有统一协议对象
- 传统系统能留痕，不能形成可验证 Proof

## 18.3 解决了通用 AI 产品做不到的问题

- 通用 AI 能理解，但不能保证合规
- 通用 AI 没有工程状态模型
- 通用 AI 不掌握项目 Fork / Override / Mapping 机制

# 20. 最终一句话结论

## 20.1 技术定义

**LayerPeg = 一个以可执行规范为核心的数据与执行协议。**

## 20.2 系统定义

**NormRef = 一个把规范、状态、证据、表单和报告统一到同一协议内的工程操作系统。**

## 20.3 产品定义

**PegBot / FormPeg / MarkUnit / PegUnit 只是这套协议栈在不同角色面前的不同入口。**

## 20.4 本质定义

**这不是“AI 问答工具”，而是“工程执行协议 + 执行操作系统”。**
